## States

| State | What user sees | Triggered by |
|---|---|---|
| Empty | Dropzone with "Drop PDFs here" message | App load, or after reset |
| Validating | Brief check after files dropped | File drop or selection |
| Files added | List of selected files, quality selector, Compress button | Valid files dropped |
| Compressing | Progress bar, "X of Y" indicator | Compress button click |
| Results | Before/after sizes, Download button | Compression complete |
| Error | Error message + Try again button | Any failure |

## Key transitions

- **Empty вҶ’ Files added:** User drops or selects valid PDFs
- **Files added вҶ’ Empty:** User removes all files
- **Files added вҶ’ Compressing:** User clicks Compress
- **Compressing вҶ’ Results:** Backend returns compressed files
- **Compressing вҶ’ Error:** Backend fails or network issue
- **Results вҶ’ Empty:** User clicks "Compress more"
- **Error вҶ’ Files added:** User clicks "Try again"

## Edge cases to handle

- User drops non-PDF files вҶ’ reject with error, keep existing valid files
- User drops 11+ files вҶ’ only accept first 10, show warning
- User drops file > 50MB вҶ’ reject that file, keep others
- User closes tab during compression вҶ’ orphaned files on server (cleanup job handles)
- Backend takes > 30 seconds вҶ’ keep showing progress, don't time out frontend prematurely
